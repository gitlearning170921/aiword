[标题2] 目的
[Body Text First Indent 2] 本规范旨在建立软件团队研发流程质量合规管理规范，为技术研发相关团队提供研发过程指导，从而保证医疗软件产品研发质量和合规性。
[标题2] 适用范围
[Body Text First Indent 2] 本规范适用于软件过程中所有活动，即适用于参与项目的所有人员。

[标题2] 研发质量管控
[标题2] 物料进货可行性研究
[小型列表] 分子公司接收到市场或者客诉等渠道的需求变更时，需求部门需要发起OA变更申请。涉及到软件变更时，流程需要流转到管理者代表审批。
[小型列表] 分子公司立项变更时，需至少提前15天向软件研发部门发起变更告知，填写《变更申请单》，交付算法包或其他物料时需同步提供《出厂测试报告》《进货检验作业指导书》，测试人员在此周期内对算法包版本、功能及场景进行全检并形成《进货检验记录》，研发人员同步进行分析并形成《》。
[小型列表] 研发部验收检验过程中，分子公司对算法包或其他物料有更新版本时，需要对算法包、SDK或其他物料进行小版本升级，同步对交付文档进行修订升级，以便区分不同的版本，防止版本被混淆使用。
[小型列表] 检验通过且经研究可行后，软件研发部门批准此变更，进行立项开发。

[标题2] 要求
[小型列表] CGM项目需要补全由0到1的初版。注册项目初版（如V1.0.0.0）按照注册文档归档，后续版本按照本规范归档。未注册项目初版按照本规范归档。
[小型列表] 医疗软件项目严格执行本规范，其中《变更申请单》、《软件需求规范》、《架构设计规范》、《系统测试方案》需要组织评审会议，会后输出评审记录并手签归档；
[小型列表] 文档不可留白，格式整齐统一、内容正确；
[小型列表] 涉及到新需求增加、硬件变更、算法变更、架构变更、个人信息收集变更、传输协议及传输内容变更时：
[小型列表] 《软件需求规范》风险评估、个人隐私评估需根据实际发版内容填写，不可填无；
[小型列表] 《架构设计规范》中网络安全评估、风险评估、个人隐私评估需根据实际发版内容填写，不可填无；
[小型列表] 初始版本所有文档涉及到的信息安全、隐私安全、网络安全需要编写覆盖齐全；
[小型列表] 《软件需求规范》《架构设计规范》《系统测试方案》需要实现需求、开发、测试之间的追溯覆盖；
[小型列表] 安耐糖APP发布上线前填写打印《软件发布说明》，产品负责人、开发负责人、测试负责人、项目经理签字确认。正式发布时填写《发布记录》，后交由QC进行检验；QC检验通过后签署《检验记录》，检查发布结果；发布成功后，测试人员进行测试验证。
[小型列表] 每次版本变更文档变更申请单、软件需求规范、架构设计规范、系统测试方案、软件发布说明、发布记录、检验记录需要立即输出，其他文档最晚在上线后1周内补齐，由于特殊原因1周内不能补齐需与管理者代表申请特批，特批后的文档最晚在上线后2周内补齐。
[标题2] 审查机制
[小型列表] 版本上线后2周内，质量部针对每个版本的完成质量和期限进行抽查核对，提出整改项；
[小型列表] 项目组成员1周内完成整改项复核及修改；
[小型列表] 质量部对检查结果进行统计汇总归档。

[Normal] 本规范从2023年6月1日开始试运行，2023年6月12日开始正式运行。
